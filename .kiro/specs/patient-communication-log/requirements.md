# Requirements Document

## Introduction

This feature adds a patient communication log to the Health Watchers platform. Clinics communicate with patients via SMS, WhatsApp, email, phone calls, and in-person interactions for appointment reminders, test results, and follow-up instructions. Tracking these communications in the patient record ensures continuity of care, provides a complete communication history, and supports compliance and audit requirements. The feature includes a data model for storing communication records, REST API endpoints for logging and retrieving communications, stub endpoints for Twilio-based SMS and WhatsApp sending, a communication timeline on the patient detail page, and inclusion of communication history in patient PDF exports.

## Glossary

- **CommunicationLog**: A record of a single communication event between a clinic and a patient, stored in the `CommunicationLog` collection.
- **Communication_Channel**: One of `sms`, `whatsapp`, `email`, `phone_call`, or `in_person`, identifying the medium used for the communication.
- **Communication_Direction**: Either `outbound` (clinic to patient) or `inbound` (patient to clinic).
- **Communication_Status**: One of `sent`, `delivered`, `failed`, or `read`, reflecting the delivery state of the communication.
- **TwilioMessageSid**: The unique message identifier returned by the Twilio API for SMS and WhatsApp messages, used for delivery status tracking.
- **Communication_Logger**: The API component responsible for receiving and persisting manual communication log entries.
- **Communication_Lister**: The API component responsible for returning paginated communication history for a patient.
- **SMS_Sender**: The API stub component responsible for accepting SMS send requests and returning a 501 response until Twilio is configured.
- **WhatsApp_Sender**: The API stub component responsible for accepting WhatsApp send requests and returning a 501 response until Twilio is configured.
- **Communication_Timeline**: The frontend component that renders the chronological list of communication events on the patient detail page.
- **Authorized_Role**: A user with the `DOCTOR`, `NURSE`, or `CLINIC_ADMIN` role.
- **Audit_Log**: The immutable record of system actions written via the existing `auditLog` service.
- **PDF_Export**: The patient health record PDF generated by the existing `sendPatientPdf` function in `export.service.ts`.

## Requirements

### Requirement 1: CommunicationLog Data Model

**User Story:** As a developer, I want a well-defined CommunicationLog data model, so that all patient communication events are stored consistently and can be queried by patient or clinic.

#### Acceptance Criteria

1. THE CommunicationLog SHALL include a `patientId` field referencing a `Patient` document (required).
2. THE CommunicationLog SHALL include a `clinicId` field referencing a `Clinic` document (required).
3. THE CommunicationLog SHALL include a `sentBy` field referencing the `User` who logged or sent the communication (required).
4. THE CommunicationLog SHALL include a `channel` field accepting exactly one of: `sms`, `whatsapp`, `email`, `phone_call`, or `in_person` (required).
5. THE CommunicationLog SHALL include a `direction` field accepting exactly one of: `outbound` or `inbound` (required).
6. THE CommunicationLog SHALL include a `content` field storing the message text or call summary as a string (required).
7. THE CommunicationLog SHALL include a `status` field accepting exactly one of: `sent`, `delivered`, `failed`, or `read` (required).
8. THE CommunicationLog SHALL include a `sentAt` date field recording when the communication was sent or logged (required).
9. THE CommunicationLog SHALL include a `deliveredAt` date field recording when delivery was confirmed (optional).
10. THE CommunicationLog SHALL include a `readAt` date field recording when the message was read (optional).
11. THE CommunicationLog SHALL include a `relatedEncounterId` field referencing an `Encounter` document (optional).
12. THE CommunicationLog SHALL include a `twilioMessageSid` string field for tracking SMS and WhatsApp messages via Twilio (optional).

### Requirement 2: Log Communication Endpoint

**User Story:** As a clinic staff member, I want to log a communication event for a patient, so that the patient record contains a complete history of all interactions.

#### Acceptance Criteria

1. WHEN a `POST /api/v1/patients/:id/communications` request is received with a valid body, THE Communication_Logger SHALL create a new CommunicationLog document and return it with a 201 status.
2. WHEN the `patientId` in the URL does not correspond to a patient belonging to the authenticated clinic, THE Communication_Logger SHALL return a 404 response.
3. WHEN the request body is missing any required field (`channel`, `direction`, `content`, `status`, `sentAt`), THE Communication_Logger SHALL return a 400 response with a descriptive validation error.
4. WHEN the `channel` value is not one of `sms`, `whatsapp`, `email`, `phone_call`, or `in_person`, THE Communication_Logger SHALL return a 400 response.
5. WHEN the `direction` value is not one of `outbound` or `inbound`, THE Communication_Logger SHALL return a 400 response.
6. WHEN the `status` value is not one of `sent`, `delivered`, `failed`, or `read`, THE Communication_Logger SHALL return a 400 response.
7. THE Communication_Logger SHALL set the `sentBy` field to the authenticated user's ID, regardless of any value provided in the request body.
8. THE Communication_Logger SHALL set the `clinicId` field to the authenticated user's clinic ID, regardless of any value provided in the request body.
9. THE Communication_Logger SHALL require the requesting user to have an Authorized_Role.

---

### Requirement 3: List Communications Endpoint

**User Story:** As a clinic staff member, I want to retrieve the communication history for a patient, so that I can review all past interactions before a consultation.

#### Acceptance Criteria

1. WHEN a `GET /api/v1/patients/:id/communications` request is received, THE Communication_Lister SHALL return all CommunicationLog entries for the specified patient scoped to the authenticated clinic.
2. WHEN the `patientId` in the URL does not correspond to a patient belonging to the authenticated clinic, THE Communication_Lister SHALL return a 404 response.
3. THE Communication_Lister SHALL support `page` and `limit` query parameters for pagination, with a default `page` of 1, a default `limit` of 20, and a maximum `limit` of 100.
4. THE Communication_Lister SHALL sort results by `sentAt` descending (most recent first).
5. THE Communication_Lister SHALL support an optional `channel` query parameter to filter results by Communication_Channel.
6. THE Communication_Lister SHALL support an optional `direction` query parameter to filter results by Communication_Direction.
7. THE Communication_Lister SHALL return a response body containing a `data` array of CommunicationLog entries and a `meta` object with `total`, `page`, and `limit` fields.
8. THE Communication_Lister SHALL require the requesting user to have an Authorized_Role.

---

### Requirement 4: SMS Send Endpoint (Stub)

**User Story:** As a clinic staff member, I want to send an SMS to a patient, so that I can deliver appointment reminders and follow-up instructions directly from the platform.

#### Acceptance Criteria

1. WHEN a `POST /api/v1/patients/:id/send-sms` request is received, THE SMS_Sender SHALL return a 501 status with a JSON body `{ "status": "error", "message": "SMS sending is not yet configured. Please configure Twilio to enable this feature." }`.
2. THE SMS_Sender SHALL require the requesting user to have an Authorized_Role.
3. THE SMS_Sender SHALL NOT create any CommunicationLog document when returning a 501 response.

---

### Requirement 5: WhatsApp Send Endpoint (Stub)

**User Story:** As a clinic staff member, I want to send a WhatsApp message to a patient, so that I can communicate through the patient's preferred messaging channel.

#### Acceptance Criteria

1. WHEN a `POST /api/v1/patients/:id/send-whatsapp` request is received, THE WhatsApp_Sender SHALL return a 501 status with a JSON body `{ "status": "error", "message": "WhatsApp sending is not yet configured. Please configure Twilio to enable this feature." }`.
2. THE WhatsApp_Sender SHALL require the requesting user to have an Authorized_Role.
3. THE WhatsApp_Sender SHALL NOT create any CommunicationLog document when returning a 501 response.

---

### Requirement 6: Communication Timeline Frontend

**User Story:** As a clinic staff member, I want to view a chronological timeline of all communications with a patient, so that I can quickly understand the history of interactions before a consultation.

#### Acceptance Criteria

1. WHEN the patient detail page is displayed, THE Communication_Timeline SHALL render a "Communications" tab alongside the existing patient detail tabs.
2. WHEN the "Communications" tab is active, THE Communication_Timeline SHALL display all communication entries for the patient in reverse chronological order, showing `channel`, `direction`, `status`, `sentAt`, and a truncated preview of `content` for each entry.
3. WHEN the Communication_Timeline is loading data, THE Communication_Timeline SHALL display a loading indicator.
4. WHEN no communication entries exist for the patient, THE Communication_Timeline SHALL display an empty state message.
5. THE Communication_Timeline SHALL support pagination, displaying a "Load more" control when additional entries are available.

---

### Requirement 7: Manual Communication Logging

**User Story:** As a clinic staff member, I want to manually log a phone call or in-person interaction, so that all patient communications are captured in the record regardless of channel.

#### Acceptance Criteria

1. WHEN the "Communications" tab is active, THE Communication_Timeline SHALL render a "Log Communication" button.
2. WHEN the "Log Communication" button is clicked, THE Communication_Timeline SHALL display a form with fields for `channel` (dropdown: all five options), `direction` (dropdown: `outbound` / `inbound`), `content` (textarea), `status` (dropdown), `sentAt` (date-time picker), and an optional `relatedEncounterId` (text input).
3. WHEN the form is submitted with valid data, THE Communication_Timeline SHALL call `POST /api/v1/patients/:id/communications` and add the new entry to the timeline without a full page reload.
4. WHEN the form is submitted with missing required fields, THE Communication_Timeline SHALL display inline validation errors and SHALL NOT submit the request.
5. WHEN the form submission fails due to a server error, THE Communication_Timeline SHALL display an error message.

---

### Requirement 8: Communication History in Patient PDF Export

**User Story:** As a clinic administrator, I want the patient PDF export to include the communication history, so that printed records contain a complete picture of patient interactions.

#### Acceptance Criteria

1. WHEN a patient PDF export is generated, THE PDF_Export SHALL include a "Communication History" section listing all CommunicationLog entries for the patient.
2. WHEN rendering each communication entry in the PDF, THE PDF_Export SHALL display `sentAt`, `channel`, `direction`, `status`, and `content` for each entry.
3. WHEN no communication entries exist for the patient, THE PDF_Export SHALL display "No communications on record." in the Communication History section.

---

### Requirement 9: Role-Based Access Control

**User Story:** As a system administrator, I want only authorized roles to access communication history, so that patient communication data is protected from unauthorized access.

#### Acceptance Criteria

1. WHEN a request to any communication endpoint is received without a valid authentication token, THE Communication_Logger SHALL return a 401 response.
2. WHEN a request to any communication endpoint is received from a user without an Authorized_Role, THE Communication_Logger SHALL return a 403 response.
3. WHEN a request to any communication endpoint is received from a user whose `clinicId` does not match the patient's `clinicId`, THE Communication_Logger SHALL return a 404 response.

---

### Requirement 10: Communication Content Privacy in Audit Logs

**User Story:** As a compliance officer, I want communication message content excluded from audit logs, so that sensitive patient communications are not duplicated in the audit trail.

#### Acceptance Criteria

1. WHEN a CommunicationLog entry is created, THE Audit_Log SHALL record the action with `resourceType: 'CommunicationLog'`, `resourceId`, `userId`, and `clinicId`, but SHALL NOT include the `content` field in the audit log `metadata`.
2. WHEN a CommunicationLog list is retrieved, THE Audit_Log SHALL record the access event with `resourceType: 'CommunicationLog'` and `patientId` in metadata, but SHALL NOT include any message content.
